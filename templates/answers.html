Question 2.2

There are a number of security issues with webSockets, such as no authentication during the handshake process, un-encrypted TCP channels which can result in sensitive data exposure, vulnerability to input data attacks which can greatly damage a website, and many others. I have chosen to focus on using a restricted payload size by limiting the size of the data which can be sent over the websocket. My reason for doing this is to prevent large payload HTTP DDoS attacks, as having a reduced payload size directly minimizes the possibility of large packets of data being sent over the websocket, such as large malware attacks. My send security feature was to enable CORS, which helps to protect against unwanted domains accessing the server. CORS explicity states which domains are allowed access and blocks calls from domains which haven't been given access, lets say an unwanted domain tried to access the server and attack our server, CORS would prevent this from happening. This makes our server more secure because we have more control over who can access it, in addition to the payload limiting, I feel that these two security measures provide a good level of security for a websocket application.
